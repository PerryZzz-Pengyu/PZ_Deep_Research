# PZ Deep Research 产品文档

## 文档维护规则

这份文档用于记录产品方向、用户定位、核心功能和体验要求。只要产品目标、用户场景、MVP 功能、页面设计、商业化方向或用户体验要求发生变化，都需要同步更新本文档。

## 产品愿景

PZ Deep Research 是一个网页端 AI 深度研究助手，面向需要可靠信息、来源证据和结构化分析的 C 端用户。

它不应该只是一个普通聊天框，而应该像一个认真做功课的研究助理：能够主动搜索资料、访问来源、整理证据、比较信息，并生成一份清晰可读的研究报告。

## 目标用户

- 学生：用于复杂主题学习、资料整理、论文前期调研。
- 职场用户：用于行业分析、竞品比较、公司研究、政策理解。
- 内容创作者：用于文章、视频脚本、播客、 newsletter 的资料准备。
- 普通用户：用于需要来源和证据的深度问题，而不是只要一句快速回答。

## 核心用户流程

1. 用户输入一个研究问题。
2. 用户选择研究模式：
   - 快速：联网搜索 -> 访问 -> 出结果。使用 1 个高命中英文搜索词，访问 3 个关键来源，输出 500 字以内 essay 风格报告。
   - 深度：联网搜索 -> 访问 -> 深度思考 -> 出结果。使用 3 个高命中英文搜索词，访问 10 个关键来源，输出 1500 字以内文献综述风格报告。
   - 专家：联网搜索 -> 访问 -> 深度思考 -> 初步研究 -> 审查缺口 -> 再次搜索 -> 再次访问 -> 最终报告。每次搜索使用 5 个高命中英文搜索词，总共访问 20 个关键来源，输出 3000 字以上论文风格报告。
3. 系统创建研究任务。
4. 用户看到实时进度：
   - 正在理解问题
   - 正在搜索资料
   - 正在访问网页
   - 正在提取证据
   - 正在撰写报告
5. 用户获得最终报告：
   - 核心结论
   - 详细分析
   - 来源列表
   - 不确定性和注意事项
6. 用户可以保存、重新运行、导出或继续追问。

## MVP 功能

- 一个研究问题输入页面。
- 研究模式选择：快速、深度、专家共用证据优先的固定流程，但使用不同搜索数量、访问数量和报告深度策略。
- 默认模型路由，后续可开放手动选择模型。
- 实时研究进度展示。
- 最终报告展示。
- 来源和引用展示：搜索得到的候选来源只在研究进度里展示，右侧来源区只展示真正访问过的来源；报告引用全部使用阿拉伯数字角标，并要求一定数量的全文证据。
- 基础历史记录。

## 后续功能

- 文件上传和私有文档研究。
- 导出 PDF 和 Markdown。
- 多模型对比。
- 定时研究更新。
- 收藏夹和资料集合。
- 付费套餐和额度控制。
- 可交互网页浏览能力。

## 产品要求

### 研究质量

- 使用网页资料时，最终答案要尽量带来源。
- 真实研究必须先搜索，再访问来源正文或可读内容，最后基于访问结果生成报告。
- 快速、深度、专家不是三套完全不同产品，而是一条固定研究流水线上的三种强度；专家模式额外包含二次搜索和补充访问。
- Agent 需要区分“来源证据”和“模型推断”。
- 如果资料互相冲突，要明确说明冲突点。
- 当证据已经足够时，不要无意义地继续搜索。

### 用户体验

- 第一屏应该直接是可用的研究界面，不做纯营销首页。
- 长任务必须展示有意义的进度。
- 用户应该能取消正在运行的任务。
- 默认展示用户能看懂的研究过程，不直接暴露复杂底层日志。

### 安全和滥用控制

- API Key 只能保存在服务端。
- 每个用户或会话需要有使用限制。
- 高风险请求需要增加内容安全检查。
- 除非有安全沙箱，否则不允许执行任意代码。

## 推荐第一屏

第一屏建议包含：

- 清晰的研究问题输入框。
- 简洁的研究模式选择。
- 可折叠的高级选项，例如模型、来源数量、报告格式。
- 最近研究记录。
- 不做大段营销文案，不做过度装饰。

## 成功指标

- 研究任务完成率。
- 首个进度事件出现时间。
- 最终报告生成时间。
- 用户导出率和复用率。
- 单次研究任务成本。
- 来源质量和引用覆盖率。

## 模型和协作使用策略

PZ Deep Research 产品本身需要支持多模型，包括 Claude、OpenAI/ChatGPT API 和 Gemini API。但项目工程实施不需要让所有模型都参与协作。

当前建议：

- Codex 负责主工程实施和本地项目维护。
- Claude Opus 4.8 作为外部架构顾问、prompt 顾问和关键评审角色。
- Gemini 暂不参与核心项目协作，优先作为产品可调用模型和后续评测对照对象。

这样做的原因是：工程实施需要持续读取本地文件、修改代码、运行测试并维护 changelog；这个角色更适合由 Codex 承担。Claude Opus 4.8 更适合在关键设计和高价值文本推理上提供第二意见。Gemini 则应该先进入产品模型能力，而不是进入项目管理链路。

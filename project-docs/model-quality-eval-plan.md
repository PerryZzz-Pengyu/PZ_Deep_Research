# PZ Deep Research — 四环节模型质量评测方案

## 文档维护规则

这份文档定义如何对研究流程的四个模型环节做任务级质量评测，用于**确定并验证云端生产路由**。
评测方式、对照集或判定标准变化时，更新本文档并记录 `project-docs/changelog.md`。

## 为什么需要它

- 公开仓的生产路由现在是 `cloud-unconfigured`；云端要"卖更好的模型路由"，**必须先有验证依据**才能固定生产模型。
- 这是 **Cloud 版正式收费的前置阻塞项**；社区版不受影响（用户自带 Key、自选模型）。
- 不做"每 Provider × 每模式"的全量回归矩阵（成本高、维护重），改为**任务级**评测：每个环节单独选最合适的模型。

## 四个环节与评测维度

Runtime 是确定性编排，模型只负责四件受限任务（见 `technical-architecture.md`）。逐一评测：

1. **意图识别与追问**（暂随追问功能一起，当前后置）
   - 维度：意图判断准确率、缺失条件识别、追问必要性与精炼度、误追问率。
2. **搜索词与工具规划**
   - 维度：英文检索词命中率（带回相关来源的比例）、模式数量服从（quick/deep/expert 的 1/3/5）、结构化输出稳定性、expert 二轮补充检索的缺口覆盖。
3. **证据卡片抽取**
   - 维度：事实抽取准确率、可引用性、压缩率、证据强度判定正确性、速度、成本。
4. **最终报告撰写**
   - 维度：事实一致性（不超出证据）、引用覆盖率与编号正确性、字数服从（quick 400–500 / deep 1300–1500 / expert 3000–3500）、References 结构服从、可读性。

## 对照集（datasets）

- **规模（起步）**：每环节 15–25 个固定研究问题，覆盖科技、医学、财经、社科、时事多领域，难度分布（事实型 / 综述型 / 有争议型）。
- **冻结**：题目与参考期望集版本化保存（建议放私有评测仓或 `project-docs/private/`，避免污染公开仓、也避免被模型预训练命中后失真）。
- **可复现**：固定 mock/真实两套输入；真实检索结果需缓存快照，保证跨次评测可比。

## 候选模型

OpenAI / Claude / Gemini 各自的候选（见 `.env.example` 的 `*_MODEL_OPTIONS`）。
**评测结论、最终模型组合和路由版本只在私有 Cloud 仓维护**；公开仓只保留可替换的 Provider/版本字段。

## 判定与产出

- **判分方式**：客观项自动校验（字数、引用编号存在性、References 章节、模式数量、结构化解析成功率）；主观项（事实一致性、可读性、追问质量）用 LLM-as-judge + 人工抽检双轨，记录评分者一致性。
- **判定标准**：每环节设最低阈值（如报告环节"引用覆盖率 ≥ X%、零超证据断言"），择优选模型，并比较成本/延迟。
- **产出**：每环节一份选型结论 → 组合成一个**版本化路由**（`MODEL_ROUTING_VERSION`），写入私有 Cloud 配置；每次调整发新版本，便于质量追踪、成本分析和故障回溯。

## 与现有测试的关系

- 复用现有结构化校验（`test_selection`、报告字数/引用/References 校验、`test_evidence`）作为自动判分的一部分。
- 真实 Provider 联调做成**可选集成测试**，无 Key 时不阻塞本地与 CI（与 testing-guide 的"第三方真实 API 联调可选"一致）。

## 恢复条件与排期

- 当前**暂缓**（规划决定）。建议恢复触发条件：①社区版发布后有真实研究流量/反馈，或 ②准备启动 Cloud 收费前。
- 恢复时先做"报告撰写"与"证据抽取"两个对用户可感知质量影响最大的环节，再做"搜索规划"，"意图/追问"随追问功能一起。

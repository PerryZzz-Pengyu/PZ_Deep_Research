# PZ Deep Research 产品文档

## 文档维护规则

这份文档用于记录产品方向、用户定位、核心功能和体验要求。只要产品目标、用户场景、MVP 功能、页面设计、商业化方向或用户体验要求发生变化，都需要同步更新本文档。

每次做了实质修改，还需要同步更新 `project-docs/changelog.md`，说明修改日期、修改内容、影响文件和修改原因。

## 产品愿景

PZ Deep Research 是一个网页端 AI 深度研究助手，面向需要可靠信息、来源证据和结构化分析的 C 端用户。

它不应该只是一个普通聊天框，而应该像一个认真做功课的研究助理：能够主动搜索资料、访问来源、整理证据、比较信息，并生成一份清晰可读的研究报告。

## 领域产品策略

产品采用共享研究工作台和领域专属研究能力，不把学术、金融、社媒、行业分析和法律研究当作只替换 Prompt 的同一流程。任务、历史、进度、来源、导出和账号体验共享；数据源、证据 Schema、研究策略、报告格式和验证规则归各领域所有。

当前状态：

- `academic` 为已实现且默认的研究领域。
- 通用任务模型和存储已具备 `domain`，路由层已通过领域注册表解析 Runtime。
- `finance` 确定为下一个领域，首发美股。其 PRD 见 `project-docs/finance-research-prd.md`，当前尚未开放 API 或前端入口。
- 多领域迁移顺序见 `project-docs/multi-domain-refactor-plan.md`。

## 目标用户

- 学生：用于复杂主题学习、资料整理、论文前期调研。
- 职场用户：用于行业分析、竞品比较、公司研究、政策理解。
- 内容创作者：用于文章、视频脚本、播客、 newsletter 的资料准备。
- 普通用户：用于需要来源和证据的深度问题，而不是只要一句快速回答。

## 核心用户流程

1. 用户输入一个研究问题。
2. 用户选择研究模式：
   - 快速：联网搜索 -> 访问 -> 出结果。使用 1 个高命中英文搜索词，最终选择 3 个关键来源，输出正文 350-900 字的 essay 风格报告。
   - 深度：联网搜索 -> 访问 -> 深度思考 -> 出结果。使用 3 个高命中英文搜索词，最终选择 10 个关键来源，输出正文 1100-2600 字的文献综述风格报告。
   - 专家：联网搜索 -> 访问 -> 初步证据 -> 审查缺口 -> 再次搜索 -> 再次访问 -> 最终报告。每次搜索使用 5 个高命中英文搜索词，从两轮访问并集中最终选择 20 个关键来源，输出正文 2700-5200 字的论文风格报告。
3. 系统创建研究任务。
4. 用户看到实时进度：
   - 正在理解问题
   - 正在搜索资料
   - 正在访问网页
   - 正在提取证据
   - 正在撰写报告
5. 用户获得最终报告：
   - 核心结论
   - 详细分析
   - 来源列表
   - 不确定性和注意事项
6. 用户可以从历史记录打开报告详情，并按原问题和模式重新运行；社区版 BYOK 凭据不保存，重跑或重试真实 Provider 时需要重新填写。
7. 用户可以把当前报告导出为 Markdown 文件。
8. 用户可以把终态报告导出为带任务信息、分页和页码的 A4 PDF。
9. 任务失败时，用户只看到产品化错误说明和一个“重试”按钮，不显示工程堆栈或第三方 Provider 原始报错。
10. 用户可以保持访客使用，也可以登录账号；首次登录会把当前浏览器中的匿名历史自动归并到账号，之后可以跨设备查看。
11. 后续继续补充追问和更丰富的文档模板。

## MVP 功能

- 营销落地页（`/`）：介绍产品价值、研究领域、研究流水线、三种模式与示例报告，并提供直接提问入口；提问后进入工作台自动开跑。
- 研究工作台（`/workbench`）：研究问题输入、模式选择、实时进度、报告与来源。
- 多语言界面：中 / 英切换，默认中文，以同一份词典中英对照维护。
- 研究模式选择：快速、深度、专家共用证据优先的固定流程，但使用不同搜索数量、访问数量和报告深度策略。
- 产品分社区版与云端版（open-core），由 `PZ_EDITION` 在运行时选择：
  - **社区版**（默认 `community`，开源 Apache 2.0）：单用户自托管工具，用户自选 Provider/模型并自带 API Key（BYOK），默认 SQLite + 访客，提供 Docker 一键运行。
  - **云端版**（`cloud`）：托管商业模式，使用私有版本化路由并忽略客户端 provider/模型/Key；订阅、额度、计费、多租户、部署和运营在独立私有仓库。
- 云端版 C 端不提供 Provider 或模型切换；具体模型组合、路由版本和成本参数不进入公开仓库。社区版按 `selection_enabled` 暴露 Provider/模型选择，以及模型、SerpAPI、Jina 的请求级 BYOK 输入。
- 实时研究进度展示。
- 最终报告展示。
- 来源和引用展示：搜索候选和访问过程只在研究进度里展示；右侧来源区仅展示质量优先后最终入选的 3 / 10 / 20 个来源（来源不足时降级展示实际数量）。最终来源重新连续编号，报告引用全部使用对应的阿拉伯数字角标。
- 基础历史记录。
  - 未登录时按浏览器匿名访客 ID 展示历史。
  - 登录后按 Clerk 用户 ID 展示账号历史，同一账号可以跨设备读取。
  - 点击历史任务进入报告详情，展示任务状态、研究配置、完整任务 ID、时间线、来源、报告草稿或最终报告。
  - 已结束任务可以按原问题和模式重新运行；新任务拥有独立 ID，并记录来源任务 ID。Provider、模型和路由版本作为内部可追踪元数据保存。
  - 首次登录自动认领当前浏览器匿名任务；任务归并后只属于账号，不允许其他访客或账号读取。
- Markdown 导出。
  - 当前报告为空时不可导出。
  - 导出内容与页面当前显示的原始 Markdown 一致，不重新调用模型，不改变引用和参考文献。
  - 文件使用 UTF-8 编码，文件名来自研究问题并自动清理系统非法字符。
- PDF 导出。
  - 仅在任务不再运行且已有报告时启用。
  - 后端读取当前访客拥有的数据库报告，由 Chromium 生成 A4 PDF，不依赖浏览器“打印为 PDF”。
  - 当前开发版 PDF 包含产品名、研究问题、模式、内部模型、生成时间、任务 ID、正文分页和页码；正式 C 端模板应隐藏不必要的 Provider/模型实现细节。
  - 报告原始 HTML 不执行，外部图片、脚本、字体和网络请求全部禁用。
- 失败恢复。
  - C 端错误分为网络错误、服务繁忙、任务超时、资料不足、积分不足、内容暂不支持和系统异常。
  - 可恢复错误在错误提示旁显示一个“重试”按钮；不可恢复错误只给出可理解的下一步，不展示工程细节。
  - 报告阶段失败且已有证据检查点时，重试只重新生成报告；其他阶段失败重新执行完整研究。
  - “重试”只用于失败恢复；成功或取消任务的“重新运行”仍是一次独立的新研究。

## 商业化信息边界

- 套餐、定价、单位成本、利润目标、投放和增长假设属于私有商业资料，不写入公开仓库。
- 公开产品文档只说明未来会提供额度和支付能力，不公开具体收费结构和经营参数。
- 实现时商业参数必须由服务端配置和版本化管理，不能写死在公开前端代码中。

## 产品要求

### 研究质量

- 使用网页资料时，最终答案要尽量带来源。
- 真实研究必须先搜索，再访问来源正文或可读内容，最后基于访问结果生成报告。
- 快速、深度、专家不是三套完全不同产品，而是一条固定研究流水线上的三种强度；专家模式额外包含二次搜索和补充访问。
- 快速模式报告正文为 350-900 字，深度模式为 1100-2600 字，专家模式为 2700-5200 字；References / 参考文献列表和 `[n]` 引用标记不计入正文字数。
- 全文证据用于质量优先和有界早停，不是会阻止任务完成的硬门槛。候选耗尽后必须进入选源降级和报告阶段，不能重复访问导致卡死。
- Agent 需要区分“来源证据”和“模型推断”。
- 如果资料互相冲突，要明确说明冲突点。
- 当证据已经足够时，不要无意义地继续搜索。

### 用户体验

- 落地页（`/`）负责讲清产品价值、研究领域、流水线和模式，并提供一个直接可用的提问入口；提问后进入工作台（`/workbench`）并自动开跑，让新用户一步上手。
- 工作台保持纯粹的研究体验：输入问题、选模式、看实时进度、读带引用报告。
- 界面支持中 / 英切换（默认中文），文案以同一份词典中英对照维护。
- 长任务必须展示有意义的进度。
- 用户应该能取消正在运行的任务。
- 浏览器刷新或 SSE 临时断线后，用户应该能继续查看当前任务。
- 历史报告详情和重新运行必须保持当前访客或账号隔离，不能通过任务 ID 越权读取或重跑其他主体的任务。
- 后端重启后，已完成任务、失败任务、事件和报告仍可从数据库恢复；重启时仍在排队或运行的任务会标记为中断失败，避免永久停留在“研究中”。
- 匿名访客 ID 只用于访客模式的历史分区，不等于账号或鉴权；登录请求必须由后端验签后绑定可信 `user_id`。
- 默认展示用户能看懂的研究过程，不直接暴露复杂底层日志。
- 用户错误信息不能包含 API Key、Provider 原始响应、环境变量名称、Python/JavaScript 堆栈或数据库连接信息。
- 当前已支持把匿名任务归并到 `user_id`，同一账号可跨设备读取自己的历史；退出登录不撤销任务归属。

### 安全和滥用控制

- 服务端部署 Key 只能保存在服务端。社区版 BYOK Key 只能存在于单次 HTTPS 请求和任务内存中，不得持久化、记录日志或进入 SSE。
- 每个用户或会话需要有使用限制。
- 高风险请求需要增加内容安全检查。
- 除非有安全沙箱，否则不允许执行任意代码。

## 工作台首屏

工作台（`/workbench`）空状态建议包含：

- 清晰的研究问题输入框。
- 简洁的研究模式选择。
- 可折叠的高级选项，例如来源范围、时间范围和报告格式；不向 C 端开放模型选择。
- 最近研究记录。
- 工作台本身不堆营销文案、不做过度装饰；产品价值与营销叙事交给落地页（`/`）。

## 成功指标

- 研究任务完成率。
- 首个进度事件出现时间。
- 最终报告生成时间。
- 用户导出率和复用率。
- 单次研究任务成本。
- 来源质量和引用覆盖率。
- 分阶段模型路由的质量、延迟和成本稳定性。
- 注册到首份报告完成率。
- 首份报告完成到首次付费转化率。
- 额度使用、付费留存和服务成本健康度；具体经营指标只在私有运营资料中维护。

## 模型和协作使用策略

PZ Deep Research 后台需要支持 Claude、OpenAI/ChatGPT API 和 Gemini API，以便按任务职责选择最合适模型并提供故障降级；C 端用户不直接选择 Provider 或模型。项目工程实施也不需要让所有模型都参与协作。

当前建议：

- Codex 负责主工程实施和本地项目维护。
- Claude Opus 4.8 作为外部架构顾问、prompt 顾问和关键评审角色。
- Gemini 暂不参与核心项目协作，优先作为后台候选模型和后续质量评测对照对象。

这样做的原因是：工程实施需要持续读取本地文件、修改代码、运行测试并维护 changelog；这个角色更适合由 Codex 承担。Claude Opus 4.8 更适合在关键设计和高价值文本推理上提供第二意见。Gemini 则应该先进入产品模型能力，而不是进入项目管理链路。

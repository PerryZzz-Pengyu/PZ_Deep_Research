# PZ Deep Research 产品文档

## 文档维护规则

这份文档用于记录产品方向、用户定位、核心功能和体验要求。只要产品目标、用户场景、MVP 功能、页面设计、商业化方向或用户体验要求发生变化，都需要同步更新本文档。

每次做了实质修改，还需要同步更新 `project-docs/changelog.md`，说明修改日期、修改内容、影响文件和修改原因。

## 产品愿景

PZ Deep Research 是一个网页端 AI 深度研究助手，面向需要可靠信息、来源证据和结构化分析的 C 端用户。

它不应该只是一个普通聊天框，而应该像一个认真做功课的研究助理：能够主动搜索资料、访问来源、整理证据、比较信息，并生成一份清晰可读的研究报告。

## 目标用户

- 学生：用于复杂主题学习、资料整理、论文前期调研。
- 职场用户：用于行业分析、竞品比较、公司研究、政策理解。
- 内容创作者：用于文章、视频脚本、播客、 newsletter 的资料准备。
- 普通用户：用于需要来源和证据的深度问题，而不是只要一句快速回答。

## 核心用户流程

1. 用户输入一个研究问题。
2. 用户选择研究模式：
   - 快速：联网搜索 -> 访问 -> 出结果。使用 1 个高命中英文搜索词，最终选择 3 个关键来源，输出正文 400-500 字的 essay 风格报告。
   - 深度：联网搜索 -> 访问 -> 深度思考 -> 出结果。使用 3 个高命中英文搜索词，最终选择 10 个关键来源，输出正文 1300-1500 字的文献综述风格报告。
   - 专家：联网搜索 -> 访问 -> 初步证据 -> 审查缺口 -> 再次搜索 -> 再次访问 -> 最终报告。每次搜索使用 5 个高命中英文搜索词，从两轮访问并集中最终选择 20 个关键来源，输出正文 3000-3500 字的论文风格报告。
3. 系统创建研究任务。
4. 用户看到实时进度：
   - 正在理解问题
   - 正在搜索资料
   - 正在访问网页
   - 正在提取证据
   - 正在撰写报告
5. 用户获得最终报告：
   - 核心结论
   - 详细分析
   - 来源列表
   - 不确定性和注意事项
6. 用户可以从历史记录打开报告详情，并按原问题、模式、Provider 和模型重新运行。
7. 用户可以把当前报告导出为 Markdown 文件。
8. 用户可以把终态报告导出为带任务信息、分页和页码的 A4 PDF。
9. 后续继续补充追问和更丰富的文档模板。

## MVP 功能

- 一个研究问题输入页面。
- 研究模式选择：快速、深度、专家共用证据优先的固定流程，但使用不同搜索数量、访问数量和报告深度策略。
- 默认模型路由，后续可开放手动选择模型。
- 实时研究进度展示。
- 最终报告展示。
- 来源和引用展示：搜索候选和访问过程只在研究进度里展示；右侧来源区仅展示质量优先后最终入选的 3 / 10 / 20 个来源（来源不足时降级展示实际数量）。最终来源重新连续编号，报告引用全部使用对应的阿拉伯数字角标。
- 基础历史记录。
  - 当前无登录阶段按浏览器匿名访客 ID 展示历史。
  - 点击历史任务进入报告详情，展示任务状态、研究配置、完整任务 ID、时间线、来源、报告草稿或最终报告。
  - 已结束任务可以按原问题、模式、Provider 和模型重新运行；新任务拥有独立 ID，并记录来源任务 ID。
  - 后续登录时，匿名历史应归并到账号，不允许不同访客互相读取。
- Markdown 导出。
  - 当前报告为空时不可导出。
  - 导出内容与页面当前显示的原始 Markdown 一致，不重新调用模型，不改变引用和参考文献。
  - 文件使用 UTF-8 编码，文件名来自研究问题并自动清理系统非法字符。
- PDF 导出。
  - 仅在任务不再运行且已有报告时启用。
  - 后端读取当前访客拥有的数据库报告，由 Chromium 生成 A4 PDF，不依赖浏览器“打印为 PDF”。
  - PDF 包含产品名、研究问题、模式、模型、生成时间、任务 ID、正文分页和页码。
  - 报告原始 HTML 不执行，外部图片、脚本、字体和网络请求全部禁用。

## 产品要求

### 研究质量

- 使用网页资料时，最终答案要尽量带来源。
- 真实研究必须先搜索，再访问来源正文或可读内容，最后基于访问结果生成报告。
- 快速、深度、专家不是三套完全不同产品，而是一条固定研究流水线上的三种强度；专家模式额外包含二次搜索和补充访问。
- 快速模式报告正文为 400-500 字，深度模式为 1300-1500 字，专家模式为 3000-3500 字；References / 参考文献列表和 `[n]` 引用标记不计入正文字数。
- 全文证据用于质量优先和有界早停，不是会阻止任务完成的硬门槛。候选耗尽后必须进入选源降级和报告阶段，不能重复访问导致卡死。
- Agent 需要区分“来源证据”和“模型推断”。
- 如果资料互相冲突，要明确说明冲突点。
- 当证据已经足够时，不要无意义地继续搜索。

### 用户体验

- 第一屏应该直接是可用的研究界面，不做纯营销首页。
- 长任务必须展示有意义的进度。
- 用户应该能取消正在运行的任务。
- 浏览器刷新或 SSE 临时断线后，用户应该能继续查看当前任务。
- 历史报告详情和重新运行必须保持当前访客隔离，不能通过任务 ID 越权读取或重跑其他访客的任务。
- 后端重启后，已完成任务、失败任务、事件和报告仍可从数据库恢复；重启时仍在排队或运行的任务会标记为中断失败，避免永久停留在“研究中”。
- 匿名访客 ID 只用于无登录 MVP 的历史分区，不等于账号或鉴权；公网产品必须使用登录态绑定 `user_id`。
- 默认展示用户能看懂的研究过程，不直接暴露复杂底层日志。

### 安全和滥用控制

- API Key 只能保存在服务端。
- 每个用户或会话需要有使用限制。
- 高风险请求需要增加内容安全检查。
- 除非有安全沙箱，否则不允许执行任意代码。

## 推荐第一屏

第一屏建议包含：

- 清晰的研究问题输入框。
- 简洁的研究模式选择。
- 可折叠的高级选项，例如模型、来源数量、报告格式。
- 最近研究记录。
- 不做大段营销文案，不做过度装饰。

## 成功指标

- 研究任务完成率。
- 首个进度事件出现时间。
- 最终报告生成时间。
- 用户导出率和复用率。
- 单次研究任务成本。
- 来源质量和引用覆盖率。

## 模型和协作使用策略

PZ Deep Research 产品本身需要支持多模型，包括 Claude、OpenAI/ChatGPT API 和 Gemini API。但项目工程实施不需要让所有模型都参与协作。

当前建议：

- Codex 负责主工程实施和本地项目维护。
- Claude Opus 4.8 作为外部架构顾问、prompt 顾问和关键评审角色。
- Gemini 暂不参与核心项目协作，优先作为产品可调用模型和后续评测对照对象。

这样做的原因是：工程实施需要持续读取本地文件、修改代码、运行测试并维护 changelog；这个角色更适合由 Codex 承担。Claude Opus 4.8 更适合在关键设计和高价值文本推理上提供第二意见。Gemini 则应该先进入产品模型能力，而不是进入项目管理链路。
